# Mots de passe

Ce guide documente le cycle de vie complet des mots de passe dans UTB Learn — de leur création à leur mise à jour, sur le mobile et le dashboard.

---

## Stockage

Les mots de passe ne sont **jamais stockés en clair**. Ils sont hashés via **SHA-256** avant insertion en BDD.

```ts
// server/utils/password.ts
hashPassword(password: string): string  // SHA-256 hash
verifyPassword(password, hash): boolean // SHA-256 compare
```

La table `agent` stocke uniquement `password_hash` — la valeur en clair n'est jamais persistée.

---

## Création (mot de passe initial)

Lors de la création d'un agent depuis le dashboard :

```text
POST /api/agent/create
    │
    ├── 1. Génération code_agent unique (6 chars base62)
    │         generateUniqueAgentCode() → "A3F7K2"
    │
    ├── 2. Le code_agent devient le mot de passe initial
    │         newPassword = code_agent
    │
    ├── 3. Hashage
    │         password_hash = hashPassword(code_agent)
    │
    ├── 4. Création Supabase Auth
    │         supabaseAdmin.auth.admin.createUser({
    │           email, password: code_agent,
    │           email_confirm: true
    │         })
    │
    └── 5. INSERT agent { code_agent, password_hash, ... }
```

---

## Vérification à la connexion

### Dashboard

```text
POST /api/auth/login
    │
    ├── SELECT agent WHERE email/code_agent = identifier
    └── verifyPassword(password_saisi, agent.password_hash)
              │
              ├── true  → connexion autorisée
              └── false → 401 "Mot de passe incorrect"
```

La vérification se fait **côté serveur uniquement** — le hash ne quitte jamais la BDD.

### Mobile

Le mobile utilise `supabase.auth.signInWithPassword({ email, password })` — Supabase Auth gère la vérification en interne contre le compte Auth créé lors de la création de l'agent.

## Mise à jour par l'agent (mobile)

L'agent peut changer son mot de passe depuis **Profil → Modifier**.

```text
handleSavePassword()
    │
    ├── 1. Validation client
    │         - ancien mot de passe non vide
    │         - nouveau >= 6 caractères
    │         - nouveau ≠ ancien
    │         - confirmation = nouveau
    │
    ├── 2. Vérification ancien mot de passe
    │         AuthService.signInWithEmail(agent.email, currentPassword)
    │         → tente une reconnexion Supabase Auth
    │         → échoue si mot de passe incorrect → erreur
    │
    ├── 3. Récupération token de session
    │         supabase.auth.getSession() → session.access_token
    │
    ├── 4. Mise à jour via API Supabase Auth
    │         PUT {SUPABASE_URL}/auth/v1/user
    │         Authorization: Bearer {access_token}
    │         { password: newPassword }
    │
    └── 5. UPDATE agent SET updated_at = now()
              WHERE email = agent.email
```

<Note>
  La mise à jour du mot de passe passe par l'**API Supabase Auth directement** depuis le client mobile — pas par une server route Nuxt. C'est pourquoi elle nécessite un token de session valide (`access_token`).
</Note>

## Réinitialisation par SUPERADMIN (dashboard)

Accessible uniquement aux **SUPERADMIN** via le menu `⋮` → "Réinitialiser le mot de passe".

```text
PATCH /api/agent/:id/reset-password
    │
    ├── 1. Vérification rôle SUPERADMIN
    │         session.user.role.designation !== "SUPERADMIN" → 403
    │
    ├── 2. Récupération agent (code_agent, email)
    │
    ├── 3. Nouveau mot de passe = code_agent de l'agent
    │         newPassword = agent.code_agent
    │
    ├── 4. Hashage et mise à jour BDD
    │         password_hash = hashPassword(newPassword)
    │         UPDATE agent SET password_hash, updated_at
    │
    └── 5. Log activité
              action: "agent_password_reset"
              meta: { code_agent, nom_complet }
```

Le nouveau mot de passe est affiché dans un toast :

```text
Mot de passe réinitialisé
"Le nouveau mot de passe est: A3F7K2"
```

<Warning>
  La réinitialisation met à jour `password_hash` dans la table `agent` (utilisé par le dashboard) mais **ne met pas à jour Supabase Auth** — le mot de passe Supabase Auth reste inchangé. L'agent pourra se reconnecter sur le mobile uniquement après un reset Supabase Auth séparé si nécessaire.
</Warning>

## Mot de passe oublié (mobile)

Accessible depuis l'écran de connexion → **"mot de passe oublié ?"**.

```text
Redirection vers /(auth)/forgot-password/email
    │
    ├── 1. Saisie email UTB
    ├── 2. Envoi OTP par email (Supabase Auth)
    ├── 3. Saisie OTP
    └── 4. Définition nouveau mot de passe
              supabase.auth.updateUser({ password: newPassword })
```

<Warning>
  **Limite Supabase Free Plan — emails OTP**

  Supabase limite l'envoi d'emails à **2 emails par heure** sur le plan Free (toutes opérations confondues : OTP, confirmation, invitation). Si plusieurs agents tentent de réinitialiser leur mot de passe simultanément, les demandes au-delà de cette limite seront silencieusement ignorées ou retournent une erreur.

  Pour un usage en production à UTB avec plusieurs agents, deux solutions :

  - **Configurer un serveur SMTP custom** dans Supabase → Dashboard → Authentication → Email Settings (SendGrid, Resend, etc.) — supprime la limite
  - **Migrer vers le plan Pro** de Supabase qui augmente significativement le quota d'emails

  En attendant, privilégier la **réinitialisation par SUPERADMIN** (section ci-dessus) qui ne passe pas par email et n'est pas soumise à cette limite.
</Warning>

## Résumé des acteurs

| Acteur | Action | Endpoint |
| --- | --- | --- |
| Système (création) | Génère code\_agent comme MDP initial | `POST /api/agent/create` |
| Agent (mobile) | Change son propre MDP | `PUT /auth/v1/user` (Supabase direct) |
| Agent (mobile) | Réinitialise via email OTP | Supabase Auth forgot-password |
| SUPERADMIN (dashboard) | Réinitialise au code\_agent | `PATCH /api/agent/:id/reset-password` |
